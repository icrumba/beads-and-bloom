<task type="auto">
  <name>Task 1: Claude CLI process manager and output parser</name>
  <files>
    src/lib/claude-parser.ts
    src/lib/process-manager.ts
  </files>
  <action>
    1. Create `src/lib/claude-parser.ts` -- parses Claude CLI streaming JSON output:
       - Claude CLI with `--output-format stream-json` outputs newline-delimited JSON objects
       - Each line is a JSON object with a `type` field. Key types to handle:
         - `assistant`: Contains Claude's text response chunks. Extract the text content for deriving the activity label.
         - `result`: Final message with `cost_usd` (number), `total_tokens` (number in usage), `duration_ms` (number). This is the completion signal.
         - `error`: Error occurred. Extract the error message.
       - Export a ClaudeOutputParser class:
         - Constructor takes callbacks: onProgress(data: { costUsd?: number, tokensUsed?: number, activityLabel?: string }), onComplete(data: { costUsd: number, tokensUsed: number, durationMs: number }), onError(error: string)
         - Method: feedLine(line: string) -- parses one JSON line and calls appropriate callback
         - For activity label: extract the last meaningful sentence/phrase from assistant text chunks (skip tool calls, keep descriptions). Truncate to 80 chars. Update on each assistant message.
         - For progress cost/tokens: some intermediate messages include running totals -- extract when available
       - Handle malformed JSON gracefully (log warning, skip line)

    2. Create `src/lib/process-manager.ts` -- spawns and manages Claude CLI sessions:
       - Use Node.js child_process.spawn to run `claude` CLI
       - Export a ProcessManager class (singleton):
         - `sessions`: Map<string, ChildProcess> tracking active sessions by task ID
         - `executeTask(taskId: string)`:
           a. Read task from database
           b. Update status to 'running', set startedAt, emit task:status event
           c. Spawn: `claude --output-format stream-json --verbose -p "{task.title}"` with cwd set to config.agenticOsDir
           d. Create ClaudeOutputParser instance wired to:
              - onProgress: update task in DB (costUsd, tokensUsed, activityLabel), emit task:progress event (throttle to max 1 event per second to avoid flooding SSE)
              - onComplete: update task status to 'review', set completedAt, final cost/tokens/duration, emit task:status event
              - onError: update task status to 'review', set errorMessage, emit task:status event
           e. Pipe stdout through line-by-line reader, feed each line to parser
           f. Handle process exit: if exit code !== 0 and not already handled by parser, treat as error
           g. On process exit (success or error): remove from sessions map
         - `cancelTask(taskId: string)`:
           a. Get process from sessions map
           b. Send SIGTERM, wait 5 seconds, then SIGKILL if still alive
           c. Update task status to 'backlog' (cancelled tasks go back to backlog), clear runtime fields
           d. Remove from sessions map
           e. Emit task:status event
         - `getActiveCount()`: return sessions.size
         - `cleanup()`: kill all active sessions (called on server shutdown)
       - Register process.on('exit') and process.on('SIGTERM') handlers to call cleanup()
       - IMPORTANT: The process manager runs server-side only. Use dynamic import or ensure it's never imported in client components.

    3. Handle edge cases:
       - Task already running (executeTask called twice): return early, don't double-spawn
       - Claude CLI not found in PATH: catch ENOENT spawn error, set task to error with message "Claude CLI not found. Ensure 'claude' is installed and in your PATH."
       - Process killed externally: handle 'error' event on child process
  </action>
  <verify>
    Write a manual test: create a task via API, call executeTask with its ID. Verify:
    - Claude process spawns (check `ps aux | grep claude`)
    - Task status updates to 'running' in database
    - Progress events are emitted (check SSE endpoint in another terminal)
    - On completion, task moves to 'review' with cost/token/duration populated
    If Claude CLI is not installed in the test environment, verify the ENOENT error handling works correctly.
  </verify>
  <done>
    Process manager spawns Claude CLI, parses streaming output for live metrics, handles completion (-> review) and errors (-> review with error), supports cancellation with clean process teardown, and prevents zombie processes.
  </done>
</task>

  <action>
    1. Create `src/lib/queue-watcher.ts`:
       - Listens for task:created and task:updated events on the event bus
       - When a task's status becomes 'queued': call processManager.executeTask(taskId)
       - This is what makes "drag to Queued = auto-execute" work
       - Initialize the watcher on first API request (lazy init, same pattern as db.ts)
       - Also: on startup, check for any tasks already in 'queued' status (recovery after server restart) and execute them
       - Export initQueueWatcher() function

    2. Create `src/app/api/tasks/[id]/execute/route.ts`:
       - POST: Manually trigger execution of a specific task
       - Sets status to 'queued' which triggers the queue watcher
       - Returns 200 with the task
       - Returns 409 if task is already running
       - This is a fallback endpoint -- normal flow is drag-to-queued

    3. Create `src/app/api/tasks/[id]/cancel/route.ts`:
       - POST: Cancel a running task
       - Calls processManager.cancelTask(taskId)
       - Returns 200 with updated task
       - Returns 400 if task is not running

    4. Wire queue watcher initialization:
       - Add initQueueWatcher() call to the event bus module or db module initialization so it starts when the server handles its first request
       - Ensure it only initializes once (guard with a boolean flag)
  </action>
